LangChain4j 的工具调用（Function Calling/Tools）机制是其最强大的功能之一，它允许大语言模型（LLM）动态调用开发者定义的业务逻辑 API。以下是其工作原理的详细解析：

核心原理：工具调用流程

  1. 工具注册

  2. 元数据生成
    LangChain4j 在运行时自动生成工具的 JSON Schema 描述，包括：

    • 工具名称（name）
    • 功能描述（description）
    • 参数结构（parameters 的 JSON Schema）

  3. 模型决策

    • 用户提问时，LangChain4j 将 工具描述 和 用户问题 一起发送给 LLM
    • LLM 判断是否需要调用工具：
      • 若需要 → 返回工具调用请求（而非直接回答）
      • 若不需要 → 直接生成文本回复

  4. 本地执行

与普通 API 调用的本质区别

特性 传统 API 调用 LangChain4j 工具调用
触发方式 开发者显式调用 LLM 动态决策触发
参数传递 手动构造参数对象 LLM 自动生成合规参数
执行上下文 独立 HTTP 请求 当前 JVM 进程内执行
错误处理 开发者手动处理 自动转换异常为 LLM 可读消息
适用场景 确定性业务流 需自然语言理解的灵活场景

总结：核心价值

  1. 自然语言接口
    用户用自然语言触发复杂业务逻辑（如“取消我周二的预约”）

  2. 业务逻辑解耦
    业务代码无需感知 AI 模型，只需关注功能实现

  3. 动态决策能力
    LLM 根据问题语义自动选择工具并生成合规参数

  4. 安全执行
    通过 JVM 沙箱、参数校验、会话隔离保障安全性

这种设计使 LangChain4j 成为连接大语言模型与传统业务系统的桥梁，特别适合需要自然语言交互的复杂业务场景（如医疗挂号、订单管理等）。

GC Root

1. 虚拟机栈中的引用对象（局部变量）

解释：obj 是 main() 方法的局部变量，存储在虚拟机栈中，属于 GC Root，所以它引用的 Object 不会被回收。

2. 方法区中的静态变量引用的对象

解释：staticObj 是类的静态变量，存储在方法区（元空间），属于 GC Root，所以它引用的 Object 不会被回收。

3. 方法区中的常量引用的对象

解释：CONSTANT 是 final 修饰的静态常量，存储在方法区（运行时常量池），属于 GC Root，所以它引用的 String 不会被回收。

4. 本地方法栈中 JNI 引用的对象

解释：JNI（Java Native Interface）方法中引用的 Java 对象会被 JVM 特殊管理，属于 GC Root，不会被回收。

5. 活跃线程（Thread 对象）

解释：thread 是活跃线程，属于 GC Root，所以它不会被回收。

6. Monitor Used（同步锁对象）

解释：lock 被 synchronized 使用，属于 GC Root，所以它不会被回收。

7. JVM 持有的对象（如 Class 对象）

解释：Test.class 是 JVM 管理的类对象，属于 GC Root，不会被回收。

8. 系统类加载器加载的类对象
